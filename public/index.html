What is Git?

Git is a version Control system for tracking changes in computer files

Git allows you to coordinate work on files among multiple people. Git and GitLab are different things, more on GitLab in just a bit!

Why do we need it?

Because we all make mistakes! Ever use the 'save as' feature so you can keep a copy of the original just in case? That is a form of version control.

Goals of Git

Accessible

Everyone can contribute!

Distributed

Enable large and distributed teams to collaborate

History

Every change is tracked
Version control allows you to time travel back through previous versions of your code. It also allows you to introduce changes from different people without losing that files history. Each person on the team has their own copy with all the history.

The Lingo!

Repository
(aka: repo)

The database where your files are stored and managed

Server

The computer that stores your repo

Client

The computer you use to connect to the server

Working/Local Copy

The repo files you keep on your computer to make changes

** Git Add - This adds the files to the repo to be tracked AND stages it. Staging is different than adding. Staging tells git to track the current state of the file.

** Git Commit - A commits saves the changes made to the file, not the file itself. Each commit has a 'hash' so we can track which changes were committed when and by whom.

** What is a hash? - Git uses SHA-1 hashes. An SHA-1 is an algorithm that takes some data as input and generates a unique 40 character string from it. That means each commit is 100% unique.

Nobody's perfect. Let's walk through how to handle mistakes.

Earlier, we created a text file and conmitted it to be tracked by git. Open that file using notepad/text-edit (right click=> Open with).

Whoops! We made a mistake. We didn't mean to change that file. What can we do?!

In terminal, type the command 'git checkout myfile.txt'. This will update the files in your root working directory to match the version in the index.

Go ahead and check it. Close the text file (if still open) and then reopen it. The text you typed in should be gone. But Why?

You made changes to the file but never committed them to the index. Until you commit them, they only exist locally. This is an important thing to remember as you work with git. Remember to commit your changes frequently.
Let's make a staged mistake .
							
								git add myfile.txt
Let's fix it !
							
								git reset HEAD myfile.txt
What happens if you added the file already? Remember, once a file is added, it is 'staged' in the index. That means Git is aware there has been a change added but not commited.

Let's give it a try. Open your text file again and add some text.

Next, add it to Git using 'git add'

Whoops! We didn't want to add that. How do we fix it?

Run the next command, 'git reset HEAD myfile.txt'

Go ahead and check it. Close the text file (if still open) and then reopen it. The text you typed in should be gone. But Why?

This command tells Git to throw away any uncommitted changes and set the file back to the last HEAD. HEAD is simply the name given to the commit from which the working tree's current state was initialized. (** This may take further exposition as HEAD can be a confusing idea for beginners)
Let's make a commited mistake .
							
								git add myfile.txt
								git commit -m "changing and commiting text"
Let's fix it !
							
								git log
								git revert [HASH]
What happens if you commited the file already? Remember, once a file is commited, it is logged in the index.

Let's give it a try. Open your text file again and add some text.

Next, add it using 'git add'

Next, commit it using 'git commit'

Whoops! We didn't want to add that. How do we fix it?

Run the command, 'git log'. This will output the hash# belonging to the latest commit. Remember the hash? It is the unique 40 character # Git gives to each commit. It's about to come in very handy!

Copy the commit Hash, then type in :q to exit the log.

Type in the command 'git revert' followed by the hash number you just copied. You should immediately see an output in the command line stating: 'On branch master nothing to commit, working tree clean'

What just happened? Well, we had already committed the change which meant that we could not simply reset. We had to go a step further and tell Git to revert the last commit. This essentially removes the changes from that commit and set's it back to the HEAD state before that commit was made.

Git & GitLab

Git ≠ GitLab

Git, GitLab...what's the difference?

GitLab

  • GitLab is open source, users can host Git repositories publicly and privately for free.
  • GitLab has one vision. Everyone can contribute to all digital content.
  • Open source projects host or mirror their repositories on GitLab.
  • Post your own code for others to use or contribute to.
  • Use and learn from the code in other people's repositories.
GitLab offers a host of features. As a beginner, the most important thing to know is that GitLab in an Open Source community dedicated to helping people share and learn from one anothers work.

**Ask the group if they know what Open Source means. Many may be unaware of the power the Open Source community has to help beginners and advanced users alike.

**Some may ask if GitLab is the same as GitHub, we get that a lot! Explain that they are similar in that they both alow you to host and share code.

Before we do anything, we need to set up your SSH Key. What is an SSH Key?

In order to push your changes to a GitLab server you need a secure communication channel for sharing information. The SSH protocol provides this security and allows you to authenticate to the GitLab remote server without supplying your username or password each time.

Step by step documentation here: https://docs.gitlab.com/ee/ssh/README.html#doc-nav
